🔢 The OSI Model

Tap any layer to see what it really does, what protocols live there, and what the data is called.

Mental model: Think of sending a letter. You write it (Layer 7), put it in an envelope (Layer 6), address it (Layer 5), hand it to postal worker (Layer 4), they sort it by region (Layer 3), put it in a mail truck (Layer 2), then the truck drives on roads (Layer 1). Each layer only talks to the layer directly above and below it.

The 7 Layers — Tap to Expand

7
Application
What the user/app sees
HTTP · DNS · FTP · Spotify API
What it does: This is where actual application data lives. When Spotify asks "give me this song", that HTTP request is a Layer 7 event. DNS queries are Layer 7 messages. Email (SMTP) is Layer 7.

Data unit name: Data / Message

Real analogy: The actual words you write in a letter.

Protocols here: HTTP/HTTPS, DNS, FTP, SMTP, SSH, DHCP (application-level), Spotify's custom API over HTTPS.

Key point: Layer 7 has NO idea about IP addresses or MAC addresses. It just says "I want this data". Lower layers handle delivery.
6
Presentation
Format, encrypt, compress
TLS · SSL · JPEG · JSON
What it does: Translates data into a format that both sender and receiver understand. This is where encryption (TLS/SSL) happens, where data gets compressed, where character encoding (UTF-8) is applied.

Data unit name: Data

Real analogy: The language you write the letter in (English vs French). And whether you seal it in a tamper-evident envelope (encryption).

For Spotify: When your song download uses HTTPS, TLS encryption/decryption happens here. The raw song bytes are also decoded (AAC/OGG codec processing) at this layer conceptually.

Key point: In TCP/IP real-world implementation, Layer 5 and 6 are often merged into the application. TLS is the most important real Layer 6 protocol.
5
Session
Manage conversations
TLS Handshake · RPC · NetBIOS
What it does: Establishes, manages, and terminates sessions between applications. A "session" is a persistent dialogue between two apps. This layer handles checkpointing (resume a broken transfer).

Data unit name: Data

Real analogy: A phone call — you dial, talk, then hang up. Session layer manages that call lifecycle.

For Spotify: Your login session to Spotify's servers is a Layer 5 concept. The TLS handshake that authenticates you to Spotify's CDN is Layer 5/6.

Key point: In practice, Layers 5 and 6 are handled entirely by the application. The OS network stack essentially skips to Layer 4 (TCP/UDP).
4
Transport
Reliable delivery between ports
TCP · UDP · QUIC
What it does: Gets data from one APPLICATION on one machine to the right APPLICATION on another machine. Uses ports to identify applications. TCP adds reliability (re-send lost packets). UDP is fire-and-forget.

Data unit name: Segment (TCP) / Datagram (UDP)

Real analogy: The postal tracking number. Ensures the letter arrives, and if it doesn't, it's resent.

Ports explained: Your computer runs many apps at once. Port numbers say "this packet goes to Chrome (port 443) not to Spotify (port 4070)". Ports 0-1023 are "well-known" (HTTP=80, HTTPS=443, DNS=53, DHCP=67/68, FTP=21).

TCP vs UDP:
TCP: "Did you get it? OK good. Did you get this one? Oh you didn't — here it is again." → used for Spotify song downloads (need every byte).
UDP: "Here's the data. Good luck." → used for DHCP discovery, DNS queries (fast, small messages).

QUIC: Spotify uses QUIC (HTTP/3) which is UDP-based but adds reliability in user-space. Best of both worlds.
3
Network
Routing across networks
IP · ICMP · ARP* · DHCP*
What it does: Routes packets from one network to another using IP addresses. This is the layer that makes the internet work — routers operate at Layer 3. IP addresses (192.168.1.1, 8.8.8.8) live here.

Data unit name: Packet

Real analogy: The ZIP code on the letter. Tells the postal system which city/region to route it to.

IP packet contains: Source IP, Destination IP, TTL (how many routers it can pass through), Protocol (TCP=6, UDP=17, ICMP=1).

Routing: Every router has a routing table. When a packet arrives, the router does a longest-prefix match and forwards it out the right interface. This repeats hop-by-hop until the destination is reached.

ARP at L2/L3 boundary: ARP is technically Layer 2 (uses MAC addresses) but it serves Layer 3 (resolves IP to MAC). It sits at the boundary.
2
Data Link
Node-to-node delivery on same network
Ethernet · WiFi · ARP · Switches
What it does: Delivers frames between two devices on the SAME network segment. Uses MAC addresses (48-bit hardware addresses like AA:BB:CC:DD:EE:FF). Ethernet and WiFi are Layer 2 protocols. Switches operate here.

Data unit name: Frame

Real analogy: The street address and building number. Gets the letter from the sorting center to the exact building.

MAC addresses: Every NIC has a unique MAC burned in at manufacture. Unlike IP addresses (logical, can change), MACs are hardware identifiers. Format: 6 bytes in hex, e.g. A4:83:E7:12:34:56.

Ethernet frame contains: Destination MAC, Source MAC, EtherType (what Layer 3 protocol is inside: 0x0800=IPv4, 0x0806=ARP, 0x86DD=IPv6), payload, FCS checksum.

Key insight: Layer 2 frames only travel on ONE network segment. When a packet crosses a router, the Layer 2 frame is completely rebuilt with new MACs for the next hop.
1
Physical
Bits on the wire/air
Ethernet cable · WiFi · Fiber
What it does: Converts bits (1s and 0s) into physical signals — electrical pulses on copper, light pulses on fiber, radio waves for WiFi. Purely hardware. No addressing, no error correction, just signal transmission.

Data unit name: Bits

Real analogy: The physical road/track that the mail truck drives on.

What's happening: Your NIC takes a binary frame and serializes it — sends bits one at a time as voltage changes (Ethernet) or frequency modulation (WiFi). The receiving NIC samples the signal and reconstructs the bits.

Ethernet physical: 1Gbps = 1 billion bits per second transmitted as voltage changes. Your cable's copper wires carry these signals. CAT5e can carry up to 1Gbps, CAT6 up to 10Gbps.

WiFi physical: Radio waves at 2.4GHz or 5GHz. Your router encodes bits as phase/amplitude changes in the radio wave (OFDM modulation). That's why walls reduce speed — they absorb/scatter the wave.

Encapsulation — How Layers Wrap Each Other

When data travels DOWN the stack (sending), each layer wraps the data from the layer above by adding a header. This is encapsulation.

L7 Data: "GET /song/12345 HTTP/1.1" (your Spotify request)
L4 adds TCP header: [Src Port: 52341 | Dst Port: 443 | Seq: 1001 | ACK: 500] + L7 Data
L3 adds IP header: [Src IP: 192.168.1.100 | Dst IP: 35.186.224.25 | TTL: 64] + TCP Segment
L2 adds Ethernet header: [Dst MAC: router-MAC | Src MAC: your-MAC | Type: 0x0800] + IP Packet + FCS
L1 transmits: ⚡ 010110100101... bits on the wire

On the RECEIVING end, each layer peels off its header (de-encapsulation) and passes the payload up. Router only looks at L3 header. Switch only looks at L2 header. App only sees L7 data.

📡 ARP — Address Resolution Protocol

"I have an IP. What's the MAC?" — ARP answers this question on every local network.

🤔 Why Does ARP Even Exist?

The internet uses IP addresses (like 192.168.1.1) to identify machines logically. But Ethernet hardware only understands MAC addresses (like AA:BB:CC:DD:EE:FF). ARP bridges this gap — it answers: "Who has IP 192.168.1.20? Tell me your MAC so I can send you a frame."
🌍
IP Address
Logical. Can change. Assigned by DHCP or admin. Used for routing ACROSS networks.
💾
MAC Address
Physical. Burned into NIC hardware. Never changes. Used for delivery on ONE segment only.

ARP only works on the same subnet. To reach a different network, you send to the gateway's MAC — the router then handles forwarding.

🔄 ARP in Action — Tap Each Step

Scenario: Your laptop (192.168.1.10) wants to send a packet to your router (192.168.1.1) but doesn't know its MAC.

💻 Your laptop needs to send to 192.168.1.1
What happens:IP layer wants to send a packet. Checks routing table: 192.168.1.1 is on the same /24 subnet → direct delivery, no gateway needed.
Kernel action:Calls neighbour subsystem: "Do I have a MAC for 192.168.1.1?" Checks ARP cache (ip neigh show).
Result:ARP cache MISS — no entry. Must do ARP.
OSI Layer:L3 IP triggered ARP which is L2/L3 boundary
↓
📢 ARP REQUEST broadcast sent to everyone
📦 The ARP REQUEST packet — every field:
Ethernet Dst MACFF:FF:FF:FF:FF:FF — BROADCAST. Every device on segment receives this.
Ethernet Src MACYour laptop's MAC: AA:AA:AA:AA:AA:AA
EtherType0x0806 — tells receiver "this is ARP, not IP"
Hardware Type0x0001 — Ethernet
Protocol Type0x0800 — IPv4 (we're resolving an IPv4 address)
Hardware Addr Len6 — MAC addresses are 6 bytes
Protocol Addr Len4 — IPv4 addresses are 4 bytes
OPER (Operation)1 = REQUEST (I'm asking). 2 would be REPLY.
SHA (Sender HW Addr)AA:AA:AA:AA:AA:AA — your laptop's MAC
SPA (Sender Protocol)192.168.1.10 — your laptop's IP
THA (Target HW Addr)00:00:00:00:00:00 — UNKNOWN (that's what we're asking for!)
TPA (Target Protocol)192.168.1.1 — the IP we want the MAC for
Total size28 bytes ARP payload + 14 bytes Ethernet header + 4 bytes FCS = 46 bytes (padded to 64 min)
English translation:"Hey everyone! I'm AA:AA:AA:AA (192.168.1.10). Who has 192.168.1.1? Tell me your MAC!"
↓
📻 Every device on the network receives the broadcast
Who gets it:Every device connected to the same switch/WiFi — your phone, smart TV, other laptops — ALL receive this frame because it's addressed to FF:FF:FF:FF:FF:FF.
What they do:Each device's NIC sees the broadcast MAC → passes frame up to OS. OS checks ARP protocol. Reads TPA (192.168.1.1). "Is that my IP?"
If NOT 192.168.1.1:Drop the packet. Do nothing. (But many smart devices silently update their own ARP cache with your IP→MAC mapping from SHA/SPA fields — called "learning".)
If IS 192.168.1.1 (router):Router recognizes its own IP. Prepares an ARP REPLY.
↓
📬 ARP REPLY sent back — unicast, not broadcast
📦 The ARP REPLY packet — every field:
Ethernet Dst MACAA:AA:AA:AA:AA:AA — YOUR laptop's MAC (from the request's SHA field). Unicast — only you get this.
Ethernet Src MACBB:BB:BB:BB:BB:BB — router's actual MAC
OPER (Operation)2 = REPLY
SHA (Sender HW Addr)BB:BB:BB:BB:BB:BB — router's MAC ← THIS IS THE ANSWER YOU WANTED
SPA (Sender Protocol)192.168.1.1 — router's IP
THA (Target HW Addr)AA:AA:AA:AA:AA:AA — your MAC
TPA (Target Protocol)192.168.1.10 — your IP
English translation:"Hey AA:AA:AA (you asked about 192.168.1.1) — that's me! My MAC is BB:BB:BB:BB:BB:BB."
↓
💾 ARP cache updated — MAC stored in kernel
Kernel action:Adds entry to neighbour table: 192.168.1.1 → BB:BB:BB:BB:BB:BB, state: NUD_REACHABLE
NUD_REACHABLE means:"I recently confirmed this MAC is valid." Will stay for ~30 seconds of inactivity.
Cache lifetime:After ~30s no traffic: REACHABLE → STALE. Next packet triggers a PROBE (unicast ARP). If no response → FAILED → removed.
Result:The queued IP packet (that triggered ARP) is NOW sent! Ethernet frame is built with dst=BB:BB:BB:BB:BB:BB and it flows to the router.
↓
✅ Original packet sent with correct MAC
Final frame:Eth dst: BB:BB:BB:BB (router) | Eth src: AA:AA:AA (you) | IP dst: 8.8.8.8 | TCP/UDP payload
Key point:Notice the IP destination is 8.8.8.8 (Google) but the Ethernet destination is the router's MAC. The frame goes to the router, which then strips the L2 header and routes the IP packet to the next hop.
Future packets:No ARP needed — cache is hit immediately. ARP overhead is only on the first packet to each new destination.

🔀 Special ARP Types

Gratuitous ARP = "Hey everyone, update your caches about ME"

What it is: An ARP where the sender IP and target IP are both the sender's own IP. Nobody asked for it — it's a voluntary announcement.

SPA:192.168.1.10 (your IP)
TPA:192.168.1.10 (SAME — your own IP)
Destination MAC:FF:FF:FF:FF:FF:FF (broadcast)

Why it's used:

1. IP conflict detection: When a host boots and claims an IP, it sends a GARP. If another machine replies with its own MAC → conflict detected!

2. Failover/migration: When a VM migrates to a new physical host, it sends a GARP so all devices update their cache with the new MAC for that IP. Used in HSRP, VRRP, live migrations.

3. Router failover: When a backup router takes over, it GARPs so all devices stop using the old router's MAC.

Proxy ARP = Router lies about being another device

A router with Proxy ARP enabled will reply to ARP requests on behalf of hosts in OTHER subnets. When Host A ARPs for 10.0.2.5 (different subnet), the router replies with its own MAC. Host A sends traffic to the router, which forwards it.

When used: Old VPNs, mobile/satellite links, specific ISP configs. Most modern networks disable it because it can cause routing loops.

⚠️ Proxy ARP can cause confusion — hosts end up with a MAC entry that's actually the router, not the real target. If the real host later appears on the same subnet, cached entries are wrong until ARP ages out.
⚠️ ARP has NO authentication. Any device can send a fake ARP reply and nobody can verify it.

ARP Spoofing / Poisoning attack:

1. Attacker sends a GARP claiming "IP 192.168.1.1 (your gateway) is at MAC: ATTACKER:MAC:HERE"

2. All devices on the network update their ARP cache — now all traffic meant for the gateway goes to the attacker!

3. Attacker forwards traffic (Man-in-the-Middle) while silently reading/modifying it.

Defenses:

• Dynamic ARP Inspection (DAI) on managed switches — validates ARP packets against a DHCP snooping table

• Static ARP entries — manually set permanent MAC→IP mappings for critical devices (gateway)

• HTTPS/TLS everywhere — even if traffic is intercepted, it's encrypted

• 802.1X port authentication — only authenticated devices can connect

🏷️ DHCP — Dynamic Host Configuration Protocol

How your device gets its IP address, gateway, DNS, and everything else automatically when it joins a network.

🤔 What Does DHCP Actually Give You?

When you connect to WiFi, DHCP hands your device a complete network "identity package":

📍 IP Address
e.g. 192.168.1.105 — your unique identity on this network
🔀 Subnet Mask
e.g. 255.255.255.0 — defines which IPs are "local" vs need routing
🚪 Default Gateway
e.g. 192.168.1.1 — the router to use for all non-local traffic
🌐 DNS Servers
e.g. 8.8.8.8 — who to ask when resolving domain names
⏱️ Lease Time
e.g. 86400s — how long you can keep this IP before renewing
🖥️ Hostname / NTP
Domain name, time server, TFTP server for PXE boot — all via DHCP options

OS does this:Assigns IP to interface (enp3s0 / eth0 / WiFi adapter). Adds subnet route. Sets default route via gateway. Writes DNS servers to /etc/resolv.conf (Linux) or DNS settings in registry (Windows).
Kernel calls made:RTM_NEWADDR (add IP), RTM_NEWROUTE (add routes) via netlink socket. These are the actual kernel messages from DHCP client daemon to kernel.
Next step:ARP for the gateway MAC (so packets can actually be sent). Then DNS lookup for any hostname resolution. Then actual application traffic begins.

⏱️ Lease Lifecycle — What Happens Over Time

Your IP lease isn't permanent. Here's the full lifecycle:

T=0 — DHCPACK received
Lease begins. IP configured. Lease Time = L seconds.
What's set:IP assigned. T1 timer = L × 0.5. T2 timer = L × 0.875. Lease expiry = T+L.
Example:L = 86400s (24h). T1 = 43200s (12h). T2 = 75600s (21h).
T1 (50%) — Renewal attempt
Client unicasts DHCPREQUEST directly to server that gave the lease.
Client sends:DHCPREQUEST unicast to server's IP. "Can I keep my IP a bit longer?"
If ACK:Lease renewed. New L, T1, T2 set. Back to happy state.
If no response:Server maybe down. Client waits until T2.
T2 (87.5%) — Rebinding attempt
T1 renewal failed. Client broadcasts to ANY DHCP server.
Client sends:DHCPREQUEST broadcast. ANY server can extend the lease. This is a last resort before expiry.
If ACK:Lease renewed by a (possibly different) server.
If no response:Approaching expiry. Client continues until T+L.
T=L — Lease EXPIRES
Client MUST stop using the IP. Network access lost.
What happens:Client removes IP from interface. Routing entries removed. Network connectivity drops.
Recovery:Client starts full DORA process again from DHCPDISCOVER.

🔁 DHCP Relay — For Large Networks

DHCP DISCOVER is a broadcast. Routers don't forward broadcasts between subnets by default. So how does one DHCP server serve 50 different subnets/VLANs?

Answer: DHCP Relay Agent (runs on the router/L3 switch)

Client sends DHCPDISCOVER broadcast on VLAN 10
Normal broadcast:Src: 0.0.0.0, Dst: 255.255.255.255. Reaches only devices on the same VLAN/segment.
↓
Router's DHCP Relay Agent intercepts the broadcast
Relay agent fills in giaddr:giaddr (Gateway IP Address field in DHCP) = relay's own interface IP on that VLAN, e.g. 10.10.10.1. This tells the DHCP server WHICH subnet the client is on.
Converts to unicast:Repackages as unicast UDP to DHCP server's IP (e.g. 192.168.100.5). Router can forward unicast across subnets.
↓
DHCP server allocates from correct pool using giaddr
Pool selection:Server sees giaddr = 10.10.10.1 → allocates from the 10.10.10.0/24 pool (not from 192.168.1.0/24 for example). Without giaddr, server wouldn't know which pool to use.
↓
Server replies to relay agent → relay delivers to client
Reply path:Server sends DHCPOFFER unicast back to the relay agent. Relay forwards to client (broadcast or unicast depending on BROADCAST flag).
Cisco config example:On router interface: "ip helper-address 192.168.100.5" — this one command enables DHCP relay for that interface.

🌐 DNS — Domain Name System

The internet's phone book. Turns "google.com" into "142.250.80.46". Distributed, cached, hierarchical.

🌳 The DNS Hierarchy — Tap to Explore

DNS is a tree. Every domain name is a path from leaf to root. Understanding the tree is key to understanding resolution.

🌍 Root Zone " . " — Top of the tree
What it is:The very top of DNS. Represented by a single dot ".". Every FQDN ends with this dot (www.google.com. — trailing dot = root).
Root nameservers:13 sets (a.root-servers.net through m.root-servers.net), operated by 12 organisations worldwide. They use anycast so there are actually hundreds of physical servers globally.
What they know:Only where the TLD servers are (.com, .org, .net, .io, .uk etc). They say: "I don't know google.com's IP but go ask the .com TLD servers."
Hardcoded where?Root server IP addresses are hardcoded into every DNS resolver software. They are the starting point for all recursive resolution.
↓
🏷️ TLD Servers — ".com", ".org", ".io" etc.
What they are:Top-Level Domain servers. Verisign operates .com and .net. ICANN operates .org. Each country has its own (.uk, .de, .in etc).
What they know:Which nameservers are authoritative for each second-level domain. E.g. "google.com is handled by ns1.google.com, ns2.google.com."
Response type:NS (nameserver) records pointing to the authoritative nameservers for the queried domain.
↓
📖 Authoritative Nameservers — the final answer
What they are:The servers that ACTUALLY hold the DNS records for a domain. Google runs ns1.google.com through ns4.google.com. Cloudflare runs DNS for millions of domains.
What they know:Every A, AAAA, MX, TXT, CNAME record for their domain. When they reply, they set the "Authoritative Answer" (AA) flag in the DNS header.
Record example:A record: www.google.com → 142.250.80.46, TTL=300 (cache for 5 minutes)
↓
🔄 Recursive Resolver — your middleman
What it is:A server that does the multi-step resolution on your behalf. Your router, ISP, or 8.8.8.8 (Google DNS), 1.1.1.1 (Cloudflare DNS) — these are recursive resolvers.
What it does:Receives "what's google.com?" → asks root → asks .com TLD → asks Google's auth NS → gets answer → caches it → returns to you.
Why cache?Once it resolves google.com, the next 1000 users asking the same thing get the cached answer instantly. This is why TTL matters — it controls how long the cache is valid.

🗺️ Full Resolution Journey — You Type "spotify.com"

Every single stop a DNS query makes — tap each step:

🌐 Browser checks its own DNS cache
Where:Chrome: chrome://net-internals/#dns. Firefox: about:networking#dns
What happens:Browser keeps its own separate DNS cache. If spotify.com was visited recently AND the TTL hasn't expired, it reuses the cached IP. No OS or network call needed.
If hit:→ Skip everything below. Use cached IP.
If miss:→ Continue to OS.
OSI layer:L7 Application — all happens inside the browser process
↓
🖥️ OS resolver: check /etc/hosts, then OS cache
What happens:Browser calls OS API: getaddrinfo("spotify.com"). OS reads /etc/nsswitch.conf: "hosts: files dns". First: check /etc/hosts file. Then: check OS DNS cache.
/etc/hosts:A simple text file. One line per mapping: "1.2.3.4 myserver.local". If spotify.com is in here, no DNS query is ever made. This is why ad-blockers can use /etc/hosts to block domains.
OS DNS cache:Linux: systemd-resolved cache (127.0.0.53). Windows: DNS Client service cache. If found and TTL not expired → return to browser.
OSI layer:L7 — still inside the OS, no network packet yet
↓
📤 DNS query sent to recursive resolver (e.g. 8.8.8.8)
📦 DNS Query Packet — every field:
IP Src:192.168.1.105 (your laptop's IP)
IP Dst:8.8.8.8 (Google's recursive resolver) — from /etc/resolv.conf
UDP Src Port:Random high port e.g. 54321 (so the reply comes back to you)
UDP Dst Port:53 — DNS server port
DNS Header ID:Random 16-bit e.g. 0xA1B2 — matches response to request
DNS QR flag:0 = Query (1 = Response)
RD flag (Recursion Desired):1 — "Please do recursive resolution for me"
QNAME:spotify.com — encoded as: \x07spotify\x03com\x00 (length-prefixed labels)
QTYPE:A (1) — asking for IPv4 address
QCLASS:IN (1) — Internet class
OSI layers:L7 DNS L4 UDP L3 IP L2 Ethernet
↓
🔄 Recursive resolver walks the DNS tree (cache miss)
Step 1 — Ask a root server:Resolver sends: "Who is authoritative for .com?" → Root server replies with NS records: "a.gtld-servers.net, b.gtld-servers.net..." (the .com TLD servers)
Step 2 — Ask TLD server:Resolver sends to a.gtld-servers.net: "Who is authoritative for spotify.com?" → TLD replies with NS records: "ns1.p34.dynect.net, ns2.p34.dynect.net..." (Spotify's authoritative nameservers)
Step 3 — Ask authoritative server:Resolver asks ns1.p34.dynect.net: "What's the A record for spotify.com?" → Auth server replies: "35.186.224.47, TTL=60"
Caching:Resolver caches each answer. Next time anyone asks for spotify.com, step 3 is skipped (cached for 60s). If .com TLD servers haven't changed, step 2 is skipped too (cached for hours/days).
All these are UDP packets:Each sub-query is a separate UDP/53 packet to a different server. If response exceeds 512 bytes (or EDNS0 limit), TCP is used instead.
↓
📬 DNS response arrives at your laptop
📦 DNS Response Packet:
QR flag:1 = Response
AA flag:0 = Non-authoritative (came from cache or recursive resolver, not the authoritative server directly)
RA flag (Recursion Available):1 — server supports recursive queries
RCODE:0 = NOERROR (success). 3 = NXDOMAIN (domain doesn't exist). 2 = SERVFAIL (server error).
Answer section:spotify.com, Type A, Class IN, TTL 60, RDATA: 35.186.224.47
After receiving:OS caches the answer for TTL seconds. Returns IP to browser. Browser can now make TCP connection to 35.186.224.47:443.

📋 DNS Record Types — What They Mean

A RecordMaps hostname → IPv4 address. Example: spotify.com → 35.186.224.47. This is what gets resolved when you open a website.
AAAA RecordMaps hostname → IPv6 address (4× as long). Example: spotify.com → 2a02:26f0:80:489::1ife. Modern browsers try AAAA first.
TTLHow many seconds resolvers cache this record. Spotify uses TTL=60 — short, so they can change IPs quickly for CDN routing. Google uses TTL=300 (5 min).
CNAME = Canonical NameAn alias. "www.spotify.com is actually a CNAME for spotify.map.fastly.net" — so resolving www.spotify.com leads to resolving spotify.map.fastly.net, which gives the real IP.
Why use it:CDN routing. spotify.map.fastly.net resolves differently depending on your location — giving you the closest CDN server. Spotify just points to Fastly and lets Fastly handle regional IPs.
Chain:www.spotify.com (CNAME) → spotify.map.fastly.net (CNAME) → 151.101.128.133 (A record). Each step is a separate DNS lookup.
MX = Mail ExchangePoints to mail servers for a domain. When someone emails you@spotify.com, their mail server looks up spotify.com MX records to find where to deliver email.
Priority:MX records have a priority number. Lower = higher priority. "10 aspmx.l.google.com, 20 alt1.aspmx.l.google.com" — try priority 10 first, then 20 as backup.
TXT = Text RecordArbitrary text data. Used for many verification purposes.
SPF:"v=spf1 include:amazonses.com ~all" — tells mail servers which IPs are allowed to send email from this domain. Anti-spam.
DKIM:Cryptographic public key used to verify email signatures. Prevents email spoofing.
Domain verification:Google/GitHub/Cloudflare ask you to add a TXT record to prove you own a domain (e.g. "google-site-verification=abc123").

⏱️ Full Timeline — Tap Each Phase

① Power Button Pressed L1 Physical
Hardware initialises. CPU, RAM, NIC — before any networking.
BIOS/UEFI boots:CPU starts executing firmware from ROM. POST tests RAM, CPU, detects hardware. PCI bus enumerated — NIC discovered.
NIC MAC address:BIOS/UEFI reads MAC address from EEPROM/flash on the NIC card itself. This is the permanent hardware address (e.g. A4:83:E7:12:34:56). Stored in the NIC's memory — never changes.
OS bootloader:GRUB/Windows Boot Manager loads kernel. Kernel initialises all subsystems including net/core/. Network stack is ready but no interface is UP yet.
NIC driver loaded:Kernel matches NIC's PCI vendor/device ID to driver (e.g. Intel → e1000e, Realtek → r8169, WiFi → iwlwifi). Driver registers the device, sets up DMA ring buffers (TX/RX queues). Interface exists but is DOWN — no traffic flows.
OSI:L1 Physical — hardware init. L2 Data Link — NIC driver + MAC address ready.
② WiFi Interface Brought UP L1 + L2
systemd/NetworkManager activates the WiFi interface. Physical link established.
Interface UP:NetworkManager/systemd-networkd sends SIOCSIFFLAGS (set interface flags) syscall to kernel → sets IFF_UP flag. NIC hardware activates. WiFi radio turns on.
WiFi association:wpa_supplicant scans for known SSIDs. Sends 802.11 Probe Request frames on each WiFi channel. Your router responds with Probe Response. 4-way WPA2 handshake completes. Cryptographic keys established.
Physical layer:WiFi radio starts sending/receiving at 2.4GHz or 5GHz. Autonegotiation sets bitrate (e.g. 300Mbps HT40, or 1Gbps on 5GHz with 802.11ac). For Ethernet: autoneg chooses 1Gbps full duplex.
STP (if Ethernet):Ethernet switch port may go through Spanning Tree Protocol states (Listening → Learning → Forwarding) — up to 30 seconds on older switches, near-instant with RSTP.
State now:Interface is UP and has link. But NO IP address yet. Can't send/receive IP traffic.
OSI:L1 radio/wire connected. L2 frames can now be sent/received.
③ DHCP — Get an IP Address L2+L3+L4+L7
The DORA dance. ~50ms on a fast home network.
DHCPDISCOVER:DHCP client creates raw socket (AF_PACKET — bypasses IP stack entirely since there's no IP). Sends frame: Eth dst=FF:FF:FF:FF:FF:FF, IP src=0.0.0.0, IP dst=255.255.255.255, UDP 68→67, DHCP op=1.
DHCPOFFER:Your router's built-in DHCP server responds: "You can have 192.168.1.105. Gateway=192.168.1.1. DNS=8.8.8.8. Lease=24h."
DHCPREQUEST + ACK:Client accepts. Server confirms. IP is now assigned.
Conflict check:Client sends Gratuitous ARP for 192.168.1.105. "Is anyone using this IP?" Waits ~1 second. No reply → safe to use.
Interface configured:Kernel: ip addr add 192.168.1.105/24. ip route add default via 192.168.1.1. /etc/resolv.conf updated with 8.8.8.8.
Packets on wire:4 UDP broadcast/broadcast packets (DORA). Plus 1 ARP probe. Total: ~5 frames.
OSI:L2 Ethernet frames. L3 IP (0.0.0.0→255.255.255.255). L4 UDP 68/67. L7 DHCP protocol.
④ ARP — Find the Gateway's MAC L2/L3
To send anything to the internet, we need the router's MAC. ARP finds it.
Why now:Any packet going to the internet must be physically delivered to the router first. But the router's MAC isn't known yet. Kernel checks ARP cache → miss → triggers ARP.
ARP REQUEST sent:Broadcast: "Who has 192.168.1.1? Tell 192.168.1.105." SHA=your-MAC, SPA=192.168.1.105, TPA=192.168.1.1.
ARP REPLY received:Router: "That's me! My MAC is CC:DD:EE:FF:00:11." Cache updated: 192.168.1.1 → CC:DD:EE:FF:00:11. State: NUD_REACHABLE.
Time taken:Typically <1ms on a local network. The request and reply are just 2 Ethernet frames.
Result:All future packets destined for the internet will be Ethernet-addressed to CC:DD:EE:FF:00:11 (router's MAC), with the IP header containing the actual internet destination.
⑤ You Open Spotify App L7 App
App starts, wants to connect to Spotify servers.
App initialises:Spotify process starts. Needs to connect to api.spotify.com and dealer.spotify.com (WebSocket for real-time updates, play state sync).
What it needs:IP address for api.spotify.com. Must do DNS resolution first.
OS call:Spotify calls getaddrinfo("api.spotify.com") → glibc → /etc/nsswitch.conf → /etc/hosts (no match) → systemd-resolved/DNS cache (cold cache, miss) → DNS query.
⑥ DNS Resolution — "Where is api.spotify.com?" L3+L4+L7
UDP query to 8.8.8.8. Recursive resolution through root → .com → Spotify's NS.
DNS query packet:Src: 192.168.1.105:54321, Dst: 8.8.8.8:53. UDP. DNS query for api.spotify.com, QTYPE=A.
L2 frame for this packet:Eth dst: CC:DD:EE:FF (router MAC — not 8.8.8.8's MAC, because 8.8.8.8 is not on your local network). Eth src: your MAC.
Resolver (8.8.8.8) does:Root → "go to .com TLD servers". TLD → "go to Spotify's NS (ns1.p34.dynect.net)". Auth NS → "api.spotify.com = 35.186.224.47, TTL=60".
Also resolves:accounts.spotify.com (for login), dealer.spotify.com (WebSocket), spclient.wg.spotify.com (client API), audio CDN domains like audio-fa.spotifycdn.com.
Each DNS query:~1 UDP packet out, ~1 UDP packet in. Round trip ~5-20ms for cached queries, ~100-300ms for cold recursive resolution.
OSI:L3 IP 192.168.1.105 → 8.8.8.8. L4 UDP port 53. L7 DNS protocol.
⑦ TCP Connection to Spotify Server L3+L4
3-way TCP handshake to 35.186.224.47:443.
SYN packet (you→Spotify):IP src=192.168.1.105, dst=35.186.224.47. TCP: src port=52341 (random ephemeral), dst port=443. Flags: SYN. Seq=1000. Options: MSS=1460, SACK, Window Scale.
SYN-ACK (Spotify→you):TCP: Flags SYN+ACK. Seq=5000. Ack=1001 (confirms your SYN). Server says "I'm here, here's my sequence number."
ACK (you→Spotify):TCP: Flags ACK. Ack=5001. "Got your SYN-ACK." Connection ESTABLISHED.
What TCP provides:Ordered delivery (seq numbers). Reliability (retransmit on loss). Flow control (receiver advertises window size). Congestion control (slow start, CWND).
QUIC (modern Spotify):Spotify may use HTTP/3 over QUIC instead. QUIC uses UDP but builds reliability on top. Advantage: connection establishment + TLS handshake in ONE round trip (0-RTT on resumption). Faster startup.
OSI:L3 IP routing. L4 TCP reliable transport.
⑧ TLS Handshake — Encrypt the Connection L5+L6
TLS 1.3 handshake. Establishes encrypted channel before any Spotify data is sent.
ClientHello:Spotify app sends: supported TLS versions, cipher suites (e.g. TLS_AES_256_GCM_SHA384), supported key exchange groups, random bytes, SNI extension ("api.spotify.com" — so CDN knows which cert to present).
ServerHello + Certificate:Server picks cipher suite, sends its X.509 certificate (contains public key + signature from CA like DigiCert). Also sends ServerFinished (TLS 1.3 merges several steps).
Certificate verification:Your OS checks certificate against trusted Certificate Authorities (CAs) in your OS trust store (/etc/ssl/certs on Linux). Verifies: not expired, not revoked, signed by trusted CA, hostname matches.
Key exchange (ECDHE):Both sides generate ephemeral Elliptic Curve Diffie-Hellman key pairs. Exchange public parts. Derive the same shared secret without ever transmitting it. Perfect forward secrecy: even if server's private key is stolen later, past sessions can't be decrypted.
After handshake:All traffic encrypted with AES-256-GCM. Attacker on your network sees only encrypted blobs, not Spotify requests.
OSI:L5 Session establishes the session. L6 Presentation handles encryption/decryption (TLS).
⑨ Spotify Login — HTTPS API Call L7 Application
Your credentials → Spotify auth servers → OAuth token returned.
HTTP request (inside TLS):POST /api/token HTTP/2. Headers: Content-Type: application/x-www-form-urlencoded. Body: grant_type=password&username=raj@...&password=... (all encrypted by TLS).
Spotify responds:200 OK. JSON body: { "access_token": "BQD3...", "expires_in": 3600, "refresh_token": "AQD..." }. This OAuth token is used for all subsequent requests.
Packets involved:HTTP/2 (multiplexed streams over one TCP connection). Multiple requests can be in-flight simultaneously. No head-of-line blocking per stream.
OSI:L7 HTTP/2 over L6 TLS over L4 TCP over L3 IP.
⑩ Browse + Tap Download Button L7 App
Spotify app requests song metadata + download URL from its API.
Metadata request:GET /v1/tracks/{track_id} HTTP/2. Authorization: Bearer {access_token}. Returns JSON with track name, artists, album art URL, available markets.
Download initiation:POST /melody/v1/item/file-manifest to spclient.wg.spotify.com. Response: list of CDN URLs where the encrypted audio file is hosted (e.g. audio-fa.spotifycdn.com/...).
CDN selection:Spotify uses Akamai, Fastly, or their own CDN. The closest CDN server is chosen (via CNAME + anycast DNS). New DNS lookup needed for the CDN hostname.
New TCP+TLS to CDN:Same process as steps 7+8 but to the CDN server. A new TLS session is established. Another TCP 3-way handshake. Another TLS 1.3 handshake.
⑪ Song File Downloaded — HTTPS GET All Layers
The actual song bytes flow. This is where FTP/HTTP concepts apply.
Protocol used:HTTPS (HTTP/2 or HTTP/3 over TLS) — NOT FTP. Modern apps use HTTPS for downloads. FTP would be plaintext and has no built-in encryption. SFTP/FTPS are encrypted but rarely used by consumer apps.
HTTP GET request:GET /audio-file/cdntracks/... HTTP/2. Range: bytes=0-4999999 (range requests allow resuming partial downloads).
CDN response:206 Partial Content. Content-Type: audio/ogg (Vorbis) or audio/mp4 (AAC). Content-Length: 5000000. Body: encrypted song bytes (Spotify uses its own encryption on top of TLS).
TCP segment flow:Song is ~5MB. With 1460-byte TCP MSS (max segment size), that's ~3500 TCP segments. Each segment: Eth header + IP header + TCP header + 1460 bytes data. TCP slow start → congestion window grows → reaches full bandwidth.
Packet path (one segment):CDN server → ISP backbone → your ISP → your router → WiFi → your laptop NIC → NIC driver → IP stack → TCP stack → TLS decryption → HTTP/2 parser → Spotify app → decrypted song bytes → written to disk.
Where TFTP fits (it doesn't here):TFTP (Trivial FTP, UDP port 69) is NOT used by Spotify. TFTP is used for PXE network booting — when a diskless machine boots from the network, it gets an IP via DHCP, then TFTP-downloads a boot image. No authentication, no encryption, just simple file transfer. Used in corporate imaging, router firmware updates, IP phones.
OSI — ALL LAYERS: L7: HTTPS/HTTP2 L6: TLS decryption L5: TLS session L4: TCP segments, ACKs L3: IP routing L2: Ethernet/WiFi frames L1: radio waves/electrical signals
⑫ 🎵 Song is saved. Download complete!
Decrypted, decoded, saved to local storage.
TLS decrypted:AES-256-GCM decryption layer strips TLS. Raw Spotify-encrypted bytes remain.
Spotify DRM decrypted:Spotify uses its own encryption (Widevine DRM) on top. The access token authorises decryption. Song bytes are now raw OGG Vorbis or AAC audio.
Written to disk:Stored in Spotify's local cache directory (encrypted with device key). When you play it offline, no network call is made — local file is read, decrypted, decoded to PCM audio, sent to audio driver → speakers.
Total network events:1 ARP (gateway MAC). 4 DHCP (DORA). ~5 DNS queries (Spotify domains). 2-3 TCP handshakes. 2-3 TLS handshakes. ~3500 TCP segments for the song file. ~3500 ACK packets back. Plus keepalives, WebSocket pings, etc.

📂 FTP vs HTTPS vs TFTP — When Each Is Used

Protocol Transport Port Encryption Used For
FTP TCP 21 (control), 20 (data) None (plaintext) Legacy file transfers, old web hosting
FTPS TCP 990 (implicit) / 21 (explicit) TLS Secure FTP, financial data transfers
SFTP TCP over SSH 22 SSH Secure file transfer, server management
TFTP UDP 69 None PXE boot, router firmware, IP phones
HTTPS TCP (or QUIC) 443 TLS 1.3 Everything modern — Spotify, Netflix, all web
Why Spotify uses HTTPS not FTP: HTTPS is the universal standard for secure application data transfer. It has authentication (TLS certificates), encryption (AES-256), works through firewalls (port 443 is always open), and supports HTTP/2 multiplexing (download song metadata + audio + images over ONE connection simultaneously).
